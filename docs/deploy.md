# Deploy — GitHub Pages

The public static site (see [docs/site.md](site.md)) is published to **GitHub Pages**
by the [`.github/workflows/pages.yml`](../.github/workflows/pages.yml) workflow.

## Pages URL

Once Pages is enabled (see the gate below), the site serves at:

> **https://mickywin22.github.io/azimuth/**

(GitHub project-site URL for `mickywin22/azimuth`. If the repo is later served from a
custom domain, this URL changes accordingly.)

## How it works

On every push to `main` (and on manual `workflow_dispatch`), the workflow:

1. checks out the repo,
2. installs Python 3.12 + the `site` extra (`uv pip install --system -e ".[site]"` —
   the `markdown` renderer; the rest of the build is pure-stdlib),
3. builds the site with `python scripts/build_site.py --out _site` (this also renders
   the cross-channel knowledge graph, `graph.json` + `graph.html`),
4. uploads `_site` as a Pages artifact (`actions/upload-pages-artifact@v3`),
5. publishes it (`actions/deploy-pages@v4`) to the `github-pages` environment.

The build output (`_site/`, like `site/`) is gitignored — it is regenerated by the
workflow on every run, never committed.

## READY-TO-FLIP gate (Michael's manual step)

**The repo is PRIVATE and the site is not live until Michael explicitly enables Pages.**
The workflow file existing in the repo does **not** publish anything on its own — the
`deploy` job only succeeds once GitHub Pages is turned on for the repo. Until then the
`deploy` job fails by design and nothing is exposed.

To flip the site live (Michael only, when ready):

1. Confirm the repo can serve Pages. GitHub Pages on a **private** repo requires a paid
   plan (Pro / Team / Enterprise); a private repo on the free plan cannot publish Pages.
   The alternative is making the repo public first — which is itself a separate,
   explicit decision.
2. **Settings → Pages → Build and deployment → Source: `GitHub Actions`.**
3. Re-run the `Deploy site to GitHub Pages` workflow (Actions tab → that workflow →
   *Run workflow*), or push any commit to `main`.
4. The site appears at the Pages URL above; the live URL is also shown on the workflow
   run's `deploy` job summary.

To take it back offline: Settings → Pages → set Source to *None* (or disable Pages).

## Validate the workflow locally

```bash
python -c "import yaml,sys; yaml.safe_load(open('.github/workflows/pages.yml')); print('pages.yml: valid YAML')"

# Reproduce the exact build the workflow runs
python scripts/build_site.py --out _site
```

if installed (`actionlint .github/workflows/pages.yml`).
